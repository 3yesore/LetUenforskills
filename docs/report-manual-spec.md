# Report Manual Spec / 拆解说明书规范

## 1. Purpose

The report is not a generic run summary. It is the primary reader-facing manual that explains how a specific skill repo is composed, how it is activated, how it runs, where evidence comes from, and what can be reused.

报告不是一次性摘要，而是一份“能读懂 skill 的拆解说明书”。它必须同时服务两类读者：

- **新手读者**：不了解 Agent Skill 设计，但需要快速知道这个 skill 是什么、何时使用、如何组成。
- **专业读者**：熟悉 prompt、agent、workflow、tooling，需要看到证据、边界、失败模式、复用资产和模型差异。

## 2. Reading Contract

Every report must answer these questions in this order:

1. **What is it?** Skill identity, purpose, target agents, input/output expectations.
2. **When does it run?** Trigger words, activation boundary, non-goals, handoff criteria.
3. **What is it made of?** `SKILL.md`, scripts, references, assets, schemas, examples, config.
4. **How does it run?** Workflow trace from entry to output, including decision points and fallback behavior.
5. **What evidence supports this?** Source paths, quoted evidence, deterministic extraction, reviewer audit.
6. **What can be reused?** Prompt blocks, workflow templates, resource patterns, quality checklists.
7. **How reliable is the analysis?** Missing evidence, unsupported claims, model uncertainty, quality gates.
8. **How do I apply it?** Learning path, adaptation steps, reuse constraints, next experiments.

If a section cannot answer its question, it must display the gap clearly instead of hiding it.

## 3. Section Order

The default report order is fixed to reduce reader confusion:

1. **Reader Guide / 阅读路径**
   - beginner path
   - expert path
   - review path
   - reuse path
2. **Executive Anatomy / 总览解剖**
   - skill name and source
   - intent sentence
   - target user/agent
   - confidence and review state
3. **Composition Map / 组成结构**
   - source files
   - roles of scripts/references/assets
   - resource loading strategy
4. **Activation Boundary / 触发边界**
   - when to use
   - when not to use
   - preconditions
   - expected inputs
5. **Workflow Trace / 流程还原**
   - ordered stages
   - inputs/actions/outputs per stage
   - decision/fallback points
6. **Implementation Decoder / 实现解码**
   - important code/resources
   - where behavior is encoded
   - runtime assumptions
7. **Evidence & Trust / 证据与可信度**
   - evidence map
   - reviewer audit
   - deterministic quality findings
8. **Reusable Assets / 复用资产**
   - patterns
   - templates
   - checklists
   - adaptation constraints
9. **Learning Notes / 学习说明**
   - plain-language explanation
   - advanced design observations
   - common pitfalls
10. **Appendix / 附录**
   - artifacts
   - raw JSON
   - source mapping
   - model/provider metadata

## 4. Reader Layers

Each important section should contain three layers, not three repeated paragraphs.

### 4.1 Plain Layer

Purpose: explain the idea for someone who has not designed a skill.

Rules:

- Use short sentences.
- Explain jargon on first use.
- Avoid model-internal uncertainty language unless it affects usage.
- Focus on the practical meaning of the section.

### 4.2 Technical Layer

Purpose: show structure and implementation details.

Rules:

- Use tables, cards, and trace rows.
- Name files, resource roles, workflow stages, and constraints.
- Separate observed facts from inferred interpretation.
- Link back to evidence and artifacts.

### 4.3 Reuse Layer

Purpose: turn analysis into portable assets.

Rules:

- State what can be copied.
- State what must be changed before reuse.
- State risks and assumptions.
- Provide concise templates when evidence is enough.

## 5. De-Duplication Rules

The report must avoid repeating the same claim across sections.

- Identity claims belong in **Executive Anatomy**.
- File roles belong in **Composition Map**.
- Step-by-step behavior belongs in **Workflow Trace**.
- Implementation details belong in **Implementation Decoder**.
- Quality claims belong in **Evidence & Trust**.
- Reuse recommendations belong in **Reusable Assets**.

A repeated claim is allowed only when it changes reader function. Example: a workflow step may appear in the trace and later appear as a reusable checklist item, but the second appearance must explain adaptation rather than restating the step.

## 6. Evidence Display

Evidence should be visible but not overwhelming.

Required fields for major claims:

- source path
- evidence id
- quote or summary
- confidence
- inference level
- reviewer state

UI recommendations:

- Show short evidence chips near claims.
- Open detailed evidence in expandable panels.
- Keep unsupported claims out of the main narrative.
- Keep unsupported or weak claims visible in review cards.

## 7. Model Attribution

A report generated by a real model must show model context without turning the report into a model leaderboard.

Required metadata:

- provider
- model id
- run id
- source repo URL or local path
- generated time
- meta-skills enabled/disabled
- reviewer status

If multiple models are compared, the report should show:

- agreement zones
- disagreement zones
- model-specific strengths
- unsupported-claim rate
- evidence coverage

## 8. Bilingual Rules

Default language is Chinese. English is available through the UI toggle.

- Navigation labels must be bilingual through `data-i18n`.
- Main report copy should be generated in Chinese first.
- English copy may be generated by the same analyst or a final curation pass.
- Technical literals such as file names, schema fields, model ids, and commands remain unchanged.
- Avoid direct word-by-word translation when a Chinese explanation needs different structure.

## 9. Visual Contract

The report should match the local UI and cinema/repo surfaces.

Style constraints:

- dark research-tool base
- glass cards with restrained borders
- high-quality typography, no browser-default small text
- clear section rhythm
- limited accent colors with semantic meaning
- motion used for orientation, not distraction

Hierarchy:

- page title is prominent but not oversized
- section titles use consistent gradient/accent language
- evidence and metadata are compact
- report body prioritizes readability over animation

## 10. Export Contract

The report consumes artifacts; it must not invent source facts.

Primary inputs:

- `source_snapshot.json`
- `structure_analysis.json`
- `workflow_analysis.json`
- `review_report.json`
- `quality_report.json`
- `patterns.json`
- `data/graph-data.json` when exported

Primary outputs:

- `report/index.html`
- skill detail pages when multiple skills exist
- links to `vault/`, `data/`, `graph/`, and `cinema/` surfaces
- copied canonical artifacts for audit

## 11. Acceptance Checklist

A report is acceptable when:

- A new reader can explain the skill purpose after the first two sections.
- A technical reader can locate the files and workflow path without reading raw JSON.
- Every important claim has evidence or is marked as inferred.
- Reuse advice is actionable and constrained.
- Duplicated content is minimized.
- Unsupported claims are not hidden from reviewers.
- Chinese default and English toggle both preserve layout.
