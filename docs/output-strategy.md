# Output Strategy

See also `docs/ui-strategy.md`, `docs/landing-page-spec.md`, and `docs/dashboard-roadmap.md` for the UI surfaces that will present these outputs.

Agent Skill Anatomy outputs should serve both immediate human reading and long-term reuse. The first-class output is an evidence-grounded Obsidian vault, but the same artifacts should later power visual graphs, reusable asset libraries, benchmark reports, and a public or private website.

## Principles

- Generate structured artifacts first, then render human formats from those artifacts.
- Keep output deterministic enough for diffs, reviews, and regeneration.
- Preserve evidence links from every note, graph node, report section, and reusable asset back to source files.
- Separate canonical data from presentation so Obsidian, reports, and websites can evolve independently.
- Support bilingual rendering without duplicating source-of-truth analysis.

## Obsidian

Obsidian remains the default publishing target because it supports local review, backlinks, graph navigation, and manual correction.

Recommended vault sections:

- `00 Maps`: high-level maps of content, sources, workflows, providers, and benchmark results.
- `01 Sources`: source repository notes with commits, acquisition method, and included skill roots.
- `02 Skills`: one note per skill with anatomy, workflow, evidence, risks, and reusable takeaways.
- `03 Patterns`: cross-skill patterns promoted only when evidence spans enough examples.
- `04 Reusable Assets`: scripts, templates, prompts, rubrics, and config snippets that can be reused.
- `05 Comparisons`: provider/model comparisons, benchmark summaries, and design tradeoffs.
- `99 System`: machine-readable inventories, analysis JSON, manifests, and review reports.

Obsidian notes should keep YAML frontmatter stable, use canonical English keys, and place bilingual prose in sections rather than duplicating metadata.

## Visual Graphs

Visual graph outputs should make relationships inspectable outside Obsidian.

Graph nodes:

- skills, source repositories, workflow steps, patterns, reusable assets, provider models, benchmark fixtures, and evidence files.

Graph edges:

- `defined_in`, `uses_asset`, `references`, `implements_pattern`, `similar_to`, `benchmarked_by`, `reviewed_by`, and `derived_from`.

Graph artifacts:

- `data/graph-data.json`: canonical node/edge data for tools and websites.
- `data/graph.mmd`: lightweight Mermaid render source.
- `graph.svg`: optional future static preview for reports and websites.

Current `graph-data.json` is generated by `asa export-data` and `asa export-all` from skill rows, resource roles, workflow trace rows, evidence audit rows, and reuse asset rows.

Graphs must avoid unsupported semantic edges. If a relationship is inferred, the edge should include confidence and evidence metadata.

## Reusable Assets

Reusable assets are outputs that can be copied into future skills or project templates.

Asset classes:

- `prompt_pattern`: reusable instruction or prompt structure.
- `workflow_pattern`: ordered procedure with prerequisites and stop conditions.
- `script`: source script or generated helper with provenance.
- `template`: Markdown, YAML, JSON, or code template.
- `rubric`: evaluation criteria or reviewer checklist.
- `config_snippet`: provider, benchmark, or harness configuration example.

Each asset should include:

- title, asset type, source skill, source path, evidence excerpts, license/provenance notes when known, and confidence.
- usage guidance, required inputs, expected outputs, and known limitations.
- stable ID so reports, graph nodes, and website pages can link to the same asset.

Generated assets should be suggestions unless explicitly curated by a human reviewer.

## Reports

Reports translate run artifacts into decision-ready summaries.

Report types:

- `run_summary`: what was analyzed, what passed review, what failed, and where artifacts were written.
- `quality_report`: unsupported claims, evidence gaps, schema issues, and reviewer recommendations.
- `benchmark_report`: provider/model comparisons by role, fixture, language, cost, and latency.
- `provider_report`: capability coverage, failures, context-limit risks, and routing recommendations.
- `release_report`: changes since previous benchmark, acceptance gate status, and known risks.

Reports should be available as JSON for automation and Markdown for human review. Markdown reports may be rendered into Obsidian and later into website pages.

## Future Website

A future website should publish selected vault content and reports without changing the core artifact pipeline.

Website goals:

- Browse skills, patterns, sources, providers, reusable assets, and benchmark results.
- Filter by agent type, provider, language, source repo, pattern, confidence, or review status.
- Show evidence panels beside analysis claims.
- Render interactive graphs from the canonical graph artifact.
- Provide copyable reusable assets with provenance and limitations.
- Support private/internal deployments before any public publishing workflow.

Website constraints:

- The website consumes rendered artifacts and manifests; it does not call models directly.
- Public pages must exclude secrets, private source paths, unpublished repositories, and unreviewed claims unless explicitly allowed.
- URL slugs should derive from stable artifact IDs, not display titles.
- Search indexes should include frontmatter and summaries, but avoid embedding large source excerpts.

## Publishing Pipeline

Long-term output flow:

```text
schemas + source evidence
  -> validated role artifacts
  -> render manifest
  -> Obsidian notes
  -> graph artifacts
  -> reusable asset catalog
  -> reports
  -> website bundle
```

Every downstream format should be regenerable from validated artifacts plus templates. Manual edits should either happen in curated overlay files or be imported back into structured artifacts so they are not lost on regeneration.

## Roadmap

1. Stabilize Obsidian rendering and manifests.
2. Add asset catalog generation from pattern and workflow artifacts.
3. Add graph JSON plus Mermaid preview generation.
4. Add benchmark and provider Markdown reports.
5. Add static website export that reuses the same manifests, graph data, and Markdown content.


## Multi-Output Export Commands

The project treats `runs/<run-id>/` as the source of truth and supports multiple output renderers:

```powershell
python -m asa export-report --run runs/<run-id> --output site/report
python -m asa export-vault --run runs/<run-id> --output vault
python -m asa export-all --run runs/<run-id> --output dist/<run-id>
```

Current outputs:

- `site/report/`: static report browser linked from the landing page.
- `vault/`: Obsidian-ready Markdown vault with maps, source notes, skill notes, workflow notes, pattern notes, quality notes, review notes, templates, and a manifest.
- `dist/<run-id>/`: combined distribution directory for report and vault outputs.

Planned outputs:

- `datasets/<run-id>/`: JSONL/CSV research dataset exports.
- `data/`: JSONL/CSV research tables plus `graph-data.json` and `graph.mmd` relationship exports.
- `indexes/<run-id>/`: retrieval chunks, BM25 metadata, and optional embedding vectors.
